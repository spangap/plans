# README Reform

## Current situation

This chapter inventories every README across the workspace as it stands today, grouped by role. Each entry summarises the README's own claims about the component — purpose, what it provides, and any caveats or stale-doc warnings the README itself flags. It is the baseline for deciding what to rewrite, consolidate, or retire.

Scope: 28 primary `README.md` files across 26 repos/straddles. Archived `README-old.old.md` copies (present in `hw-tdeck`, `spangap-core`, `spangap-core/esp-idf`, `spangap-web/browser`) are excluded — they are superseded snapshots.

> A cross-cutting warning already lives in the build-system README: sibling READMEs are mid-rewrite. Known rot includes `s/` paths that don't exist, aspirational file-layout boxes, per-straddle `CLAUDE.md` files superseded by `INTERNALS.md`/`docs/`, and hand-written `app_main()` init sequences that are now obsolete (init is build-generated). Treat that as the reform's starting to-do list.

### Build system & tooling

#### spangap
Repository holding the command-line launcher, installer, and build system for the Spangap platform — a Device Application Framework for the ESP32 with on-device firmware and a web interface. Firmware builds entirely inside an auto-provisioned Docker container, so the host only needs `docker`, `git`, `python3`, and the single-file `spangap` launcher (installed via a curl one-liner or dropped on `$PATH`). A persistent home lives in `~/.spangap`; everything else stays in the initialized workspace. The launcher is a thin dispatcher — the real build system lives in the per-workspace `spangap/` skeleton created by `spangap init`, and a workspace can auto-upgrade a stale launcher in place. Host prerequisites (docker for the build env; git for skeletons/straddles; python3 `venv` for pinned `esptool`/`esp-idf-monitor` serial tooling) are documented in detail. Linux and Mac supported, Windows untested. Points to INTERNALS.md and the GitHub org pages.

#### spangap/build-system (in-container map)
Container-side orientation doc for an agent running as user `spangap` inside the build Docker container: what is mounted, which `spangap` verbs work in-container vs host-only, and how to write/maintain "straddles" (the platform's unit of distributable functionality — one git repo with a `straddle.yaml` manifest plus optional firmware/`esp-idf`, browser, and on-device-LCD halves, laid out as flat polyrepo siblings). Documents the in-container CLI (`build`, `validate`, `list-requires`, `clean`, `show`, `cli`, signal-only `flash`/`reset`, `log`), the host↔container device loop driven by a host `spangap monitor` (flash signalling via `.spangap-flashme`, network CLI over ssh/TCP, log relay on port 2324), the mandatory flat workspace layout, the `straddle.yaml` schema, the prefix-vs-repo-name namespace map, and what `spangap build` does under the hood (dep resolution, staging, Kconfig gating, linting, auto-init generation of `spangapInitStraddles()`). Caveats: in-container `.spangap-log` is unreadable via `cat`/`tail` on Docker Desktop virtiofs (use `spangap log`); the running `spangap` is the baked image copy, so source edits need a rebuild; macOS `fakeowner` causes spurious git file-mode noise; and the platform realities (ESP32-S3 mandatory octal PSRAM, ESP-IDF 5.5.4, Node 20, PSRAM/LittleFS flash-cache crash hazard). Ends with the explicit "stale-doc caveats" section noted above.

### Platform core & components

#### spangap-core
Base firmware runtime of the platform — the primitives every spangap device needs regardless of the app on top — packaged as a single ESP-IDF managed component published as `spangap/spangap-core`. Its straddle `prefix` is the empty string, so its symbols (`storageGet`, `cliRegister`, `info`, `itsPoll`, `gp_alloc`, …) read as language primitives; every other firmware straddle depends on it. A multi-function straddle: init, storage, fs, its (Inter-Task Streaming IPC), logging, cli, auth, cron, power-management, memory, and idf-tweaks, each with an operator guide plus internals reference under `docs/`, alongside cross-cutting docs for flash-partitions and remote-access. Includes a source-layout table (public headers in `esp-idf/include/`, implementations in `esp-idf/src/`, one header/source pair per module) and an explicit "does NOT own" list — WiFi/TCP/TLS/NTP/mDNS (spangap-net), HTTPS/auth-enforcement/WebRTC/browser shell (spangap-web), the LVGL shell (spangap-lcd), camera/audio/app logic (the consuming app). Consumed via `spangap/spangap-core: "^0.1.0"` (or `path:` for sibling-checkout dev); its tasks start automatically while consumers compose around it via subscriptions, cron, CLI registration, boot scripts.

#### spangap-core/esp-idf
Firmware half of the spangap-core straddle — the ESP-IDF managed component in standard IDF component structure. Documents what the subdir owns (`idf_component.yml`, `CMakeLists.txt` globbing `src/*.{cpp,c}`, `Kconfig`, `project_include.cmake`, `sdkconfig.defaults.spangap`, plus `cmake/`, `data/factory_state/`, `include/`, `src/`, `scripts/`) and how a consumer pulls it in via registry dep or sibling `path:` injected through `--spangap`, layering `sdkconfig.defaults.spangap` first. Notes: all `src/` files compile as part of the component with `include/` and `src/` on the include path; public API headers live in `include/` and the CMake `REQUIRES` must cover every symbol they expose; app concerns (camera/audio/detect/recording/RTSP/AVI) don't belong here. OTA takes a public key at init (`otaInit(pubkey_pem, len)`) supplied as `OTA_PUBKEY_PEM`.

#### spangap-net
IP-networking half of the platform: WiFi, TCP/UDP, TLS, NTP, mDNS, pulled in by any straddle needing an IP stack (spangap-web, acme, duckdns, upnp, wg, sshd, ota, TCP/Auto RNS interfaces); LoRa/ESP-NOW-only nodes skip it to save lwIP and WiFi-blob RAM. Four functions — net (WiFi STA/AP state machine, TCP socket relay, `NET_EV_*` event bus, `wget` CLI), tls (shared mbedTLS server, EC P-256 cert, hot reload), ntp (non-blocking sync, timezone resolver, `date` CLI), mdns (`<hostname>.local`). Runs in the dispatcher's platform band so the stack is up before any service/app straddle; bundles remote-access add-ons (acme, upnp, wg, duckdns, sshd) by default via `additional_installs`. Owns the well-known TCP port config keys (`http_port` 80, `https_port` 443, `rtsp_port` 554, `webrtc_port` 4433, plus off-by-default `log_port`/`cli_port`). Does not own the HTTPS server/REST/WebDAV (spangap-web), the WireGuard tunnel, or the RNS interfaces. Depends transitively on spangap-core.

#### spangap-web
Web-stack half of the platform: the device's HTTPS server and file host, cookie-session auth enforcement, WebRTC plumbing for every device↔browser data path, and the shared browser UI shell. Also the browser-side UI activator — when in the build it picks up every other straddle's `browser/` subdir and folds them into one SPA. Multi-function: web (HTTP/HTTPS, file serving, WebDAV, URL-forwarding, auth, loopback exemption), webrtc (DTLS/ICE-lite/SCTP plumbing, content-free DataChannel↔ITS router), browser-shell (SPA shell with Dock launcher, `registerApp`, menu store, `GeneratedPanel`, WebRTC session, config sync, auth flow). Firmware half in `esp-idf/`; browser half is npm package `spangap-browser`. `web`/`webrtc` start automatically in the platform band after core and net; depends on spangap-net (WiFi/TCP/TLS/certs/ports) and hard-depends on upnp (external IP in SDP answers). Does not own WiFi/TCP/TLS/certs (net), the credential/session primitive (core auth), the Network/WiFi-scan panels (net's `browser/`), or camera/video/RTSP UI (the consuming app).

#### spangap-web/browser (spangap-browser)
Browser-side runtime and shared UI shell for spangap device apps — the npm half of the spangap-web straddle, paired in lockstep with its firmware half. Provides the WebRTC session manager, config sync, config-bound `Setting*` controls, the path-based menu/settings system with the `GeneratedPanel` renderer, the Dock app launcher, Floating/Log/Terminal/Editor windows, default System/About/Developer panels, and the auth/session flow plus login/setup pages. Ships TypeScript/Vue source (no `dist`) consumed directly by the consumer's bundler (Vite via Quasar); API exposed through subpath exports (`lib/*`, `components/*`, `modules/*`, `pages/*`, plus Pinia stores). Modules in `modules/*` self-register with the menu store on import, which generated `straddles.gen.ts` does automatically for every staged straddle. Conventions: config-bound `Setting*` controls (subscription/optimistic-update/rollback), `registerApp` for the Dock, menu-store registration for settings panes, `useDeviceStore()` over the `storage:1` DataChannel, singleton `webrtcSession`. Does not own camera/video/RTSP/recording UI, Network/WiFi-scan panels, or app-specific panels.

#### spangap-lcd
On-device UI for the platform: a phone-style shell (status bar, paged launcher, navigation, recents switcher) on a per-app object model, with built-in Settings, Log and CLI apps, drawn with LVGL v9 on an SPI display. The screen counterpart to the spangap-web browser SPA; owns the display, input devices, and the single LVGL context. Other straddles install apps/settings panes via the public API (`lcdInstall`, `lcdRegisterSettings`) rather than touching `lv_*`. A clean reimplementation (no surviving code) of the ESP-Brookesia phone UI, with a libvterm-backed VT100 terminal, checked-in fonts, and RGB565A8 launcher icons. Init runs automatically (`lcdInit()`) as the last platform-band entry so the display is up before other straddles; display config is Kconfig-driven (`CONFIG_LCD_*`) while input stays board-specific via an `lcd_input_t` HAL. Caveats: does not blank on inactivity (sets ephemeral `sys.standby`; standby policy is the board's), and `s.lcd.icon_res` is inert (icon size fixed to 36×36 in code).

#### viewer
Micro HTML/Markdown document viewer for spangap devices — the platform's README/help/CHANGELOG reader, named "viewer" to avoid confusion with the user's web browser. Two front ends over one behaviour: an LCD "Info" launcher app rendering a safe HTML subset into LVGL, and a Web UI "Viewer" Dock app loading a URL into an `<iframe>` served by the device's own web server. Markdown → HTML happens on the device's web server (vendored MD4C), so both front ends consume HTML, with no http(s) proxy. Owns the `viewer`/`s.viewer` config trees and the `lcdview`/`webview` CLI verbs, across always-compiled and conditional `esp-idf/` pieces plus a `browser/` Vue panel and built-in WELCOME/ABOUT pages. All `s.viewer.*` keys unset by default; `once_*` is the only auto-open mechanism (surfacing a CHANGELOG after update), `home_*` falling back to the welcome page. Soft-depends on spangap-net (http client), spangap-web (Markdown transform/serving, loopback-exempted from auth), spangap-lcd (LCD front end); spangap-core implicit. Caveats: no reverse-proxy (shelved, needs APSTA); LCD rendering omits images and styled tables and caps large documents rather than virtualizing.

### Reticulum core & interfaces

#### rns
The straddle bringing Mark Qvist's Reticulum network stack to the device, centred on `rnsd` — a single FreeRTOS task owning the entire RNS protocol state (identity, destinations, path table, Transport state machine, Links, Resources) with zero networking or radio dependencies of its own. Its protocol engine is a modified fork of `attermann/microReticulum`, adapted with device-native crypto (mbedTLS + ed25519-donna/x25519), a hand-rolled MsgPack shim, cJSON, a central allocator, and PSRAM table placement; `rnsd` bridges µR to the rest of the device over ITS. Interface straddles (iface-tcp/lora/espnow/auto) plug in over `RNSD_PORT_IFACE` as bounded packet pipes, while consumers (lxmf, nomad) operate on byte arrays via the `rnsd.h` C API, never touching µR types — an isolation letting the engine be swapped. Exposes reliable in-order Channels riding inside best-effort Links, an ITS port map (ports 1–11), and a large set of `s.rnsd.*` settings plus published `rnsd.*`/`rns.ready` telemetry. Caveats: `s.rnsd.enable` is read once at boot (reboot to change); the single-threaded task load-sheds `Transport::jobs()` sweeps under inbound bursts; full fork details in INTERNALS.md.

#### iface-tcp
Carries Reticulum traffic over plain TCP/IP — dialing outbound peers and accepting inbound connections, framing RNS packets with the same HDLC byte-stuffing as upstream `TCPInterface`, registering each connection as its own rns interface. Owns no sockets (borrows TCP from spangap-net, the protocol engine from rns); runs one task bridging two ITS services — outbound peers as `tcp/<id>`, inbound clients as `tcp_in/<addr>#<slot>` — with dropped connections deregistering and reconnecting via exponential backoff. Configuration entirely storage-driven (outbound peers in JSON array `s.tcp.peers`, inbound server/global gate under `s.tcp.*`, passphrases as on-device secrets); `s.tcp.enable` defaults to 1 (global gate), inbound server off by default (`server_port` 4965, max 8 concurrent). Supports per-interface modes and IFAC; exposes self-clearing command sentinels (`tcp.cmd.connect/disconnect/restart/del`). Inbound connection state shown only via the `tcp` CLI, not mirrored to storage.

#### iface-auto
AutoInterface for rns: zero-configuration Reticulum over a local WiFi link, wire-compatible with upstream's `AutoInterface` so a spangap device and a desktop Reticulum node on the same LAN peer with nothing configured on either side. A from-scratch C++ reimplementation reproducing upstream's group-address derivation, peering-token exchange, and well-known ports byte-for-byte; runs as two tasks (`auto` + `auto-rx`), uses IPv6 link-local multicast for peer discovery and unicast UDP for data, self-registers with rnsd as one broadcast-style interface (MTU 500, 10 Mbit/s). Depends on spangap-net for WiFi/IP; gates on `netIsUp()`. Off by default (`s.auto.enable = 0`); peers sharing the same group name (`s.auto.group`, default `"reticulum"`) appear within seconds. Supports IFAC access-coded networks (byte-compatible), credentials passed to rnsd which owns key derivation.

#### iface-espnow
RNS interface carrying Reticulum packets over Espressif's ESP-NOW link using the chip's long-range PHY, presenting a shared broadcast medium where one RNS packet maps 1:1 to one ESP-NOW frame (v2 API driver-level fragmentation lets a full 500-byte RNS packet ride one `esp_now_send`). Runs one task, registers a single broadcast peer, self-registers with rnsd as `espnow` (gateway mode, MTU 500, 250k/500k bit/s), tagging frames with a 4-byte magic prefix (`"RNS"` + framing-version byte) and dropping other ESP-NOW traffic. WiFi radio is a hard dependency owned by spangap-net; a channel-conflict policy (`s.espnow.conflict_policy`, default `disable` vs `stay`) governs behaviour when the ESP-NOW channel disagrees with an associated AP's channel, re-checked on net up/down and a throttled 30s poll (the only way to catch an AP silently moving channel). IFAC supported, passed through to rnsd.

#### iface-lora
LoRa interface for rns, carrying Reticulum packets over any RadioLib LoRa chip (SX126x, SX127x/RFM9x, SX128x, LR11x0, LR2021) and driving up to four radios off one shared SPI bus, each registering with rnsd as `lora/<slot>`. Uses jgromes/RadioLib (the `radiolib` IDF component) with a custom ESP-IDF HAL (`EspIdfHal`), running a single chip-agnostic task that dispatches only per-chip bring-up by family and frames RNS packets (MTU 500, airtime-derived bitrate). Radios configured via Kconfig (`LORA_*`/`LORAn_*`, pins/chip/TCXO/RF-switch), starting automatically when `CONFIG_LORA_COUNT > 0` (default 0 = inert, RadioLib linked out); a radio refuses to come up until `frequency`, `tx_power`, and valid SF/BW/CR/preamble are set (state reads `unconfigured`). No spangap-net dependency (bare radio, no IP). Caveat: a complex chip-DIO RF-switch table for some LR11x0/LR2021 boards isn't expressible as two pins and needs board-supplied support; also does not own the LoRa power rail or antenna selection.

### Reticulum apps & services

#### lxmf
The device's LXMF messaging mailbox — sends and receives signed LXMF messages over rns, interoperating wire-for-wire with stock clients (Sideband, NomadNet, MeshChat). A from-scratch C++ reimplementation pinned to LXMF 0.9.8, sitting above the Reticulum stack and reaching the network only through rnsd's byte-array C API. Packs/signs/verifies messages, holds up to four fully-siloed identities (`LXMF_MAX_IDENTITIES = 4`), auto-selects delivery mode (opportunistic packet, DIRECT Link, or Resource transfer), advertises identities via announces and collects a cross-identity announce catalogue, and pays/enforces proof-of-work stamps. Entire API is storage reads/writes (no DataChannel or client ITS port): clients write self-clearing command sentinels and read published state, giving multi-frontend coherence with disjoint client/firmware-owned fields. Caveats: no identity created automatically (a device with none runs transport-only), no automatic send retry, `sent` is not `delivered` (opportunistic packets get no ack), and tickets are unimplemented so `s.lxmf.auto_ticket` has no effect.

#### nomad
The Nomad Network page/"text web" client on rns, independently reimplemented in C++/TypeScript (not a port of the GPL sources), browsing Micron pages and files hosted on `nomadnetwork.node` destinations over Reticulum Links. The task is pure transport and state — never parses Micron, holds zero RNS types — riding rnsd's byte-array API (`rnsdLinkOpen`/`rnsdLinkRequest`) and using storage as its API: maintains an LRU announced-nodes feed, navigates pages, caches pages to save air time, packs/submits forms, persists bookmarks. Browsing is session-based with 7 parallel contexts (sessions 0–5 web tabs, 6 the on-device LCD), each holding its own Link and nav state; frontends drive it by writing `nomad.cmd.*` sentinels and reading `nomad.*` state. Pages under 128 KB (`NOMAD_MAX_PAGE_PUBLISH`) publish to the storage tree; larger pages set `truncated=1` with empty body and live only in the device-side cache. Explicitly client-only — no node/server hosting, no Micron rendering in firmware (frontends render), no file download, in-page partials, or `rrc://` chat.

#### rnsh
A Reticulum remote shell packaged as one straddle with two halves: a server advertising an `rnsh` destination, accepting an incoming reliable Channel and bridging it to the device `cli` with the admin-password login gate forced on; and a client (`rnsh <dest_hash>`) dialing a remote node and relaying an interactive terminal over the same Channel. The mesh-native analogue of sshd; because the Reticulum Link is already end-to-end encrypted and authenticated, rnsh implements no crypto of its own — its only security job is the application-level admin-password gate (enforced inside `cli`, three wrong tries drop the session, no command runs before login). Server off by default (`s.rnsh.server.enabled=0`, live-reconciled within ~1 s), admits no one until an admin password is set, runs up to two concurrent sessions, collapses remote stdout/stderr into one stream; admin password lives in spangap-core's `auth` `admin` realm, not an rnsh secret. Client needs no configuration or key; `..!` at line start disconnects. Does not own the Channel primitive/rnsd bridge (rns), the cli/login gate/auth realm (spangap-core), or the TCP/SSH shell (sshd).

#### maps
Offline slippy-map viewer for spangap LCD devices: an LVGL launcher app blitting pre-baked map tiles (JPEG with raw RGB565 fallback) from SD, centred on the live GPS fix, with drag-to-pan and pinch/button zoom. Nothing rendered or fetched on-device — tiles baked on a workstation (following the `esp32_offline_osm` desktop-render/device-blit split) and copied to SD at `/sdcard/maps/<z>/<x>/<y>.{jpg,bin}`, read on demand and cached in PSRAM. Independent of the RNS family (published under `reticulous/` for discoverability and as the docs' "first non-RNS feature straddle" walkthrough); on-device only, no browser UI. `mapsInit()` is a no-op and the viewer is brought up automatically via `mapsLcdRegister` only when `spangap-lcd` is staged. GPS fix arrives through ephemeral `gps.*` storage keys (no compile-time GPS dependency); zoom capped to where SD tiles exist. Ships a Docker-based `tilebake` pipeline (single-bbox and batch/layered maps). Note the OSM/ODbL attribution and share-alike obligations on redistributed tiles.

### Remote-access & networking services

#### acme
A self-contained ACMEv2 (RFC 8555) client — built on mbedTLS with ES256/JWS signing, an mbedTLS X.509 CSR, `esp_http_client` transport, and a hand-built UDP DNS resolver — obtaining and renewing publicly-trusted TLS certificates (Let's Encrypt by default). Starts automatically when in the build, registers an `acme` CLI command, seeds a daily cron entry renewing when the cert is missing, self-signed, or near expiry; renewal needs `s.acme.enable=1`, a public DNS name in `s.net.dns.fqdn`, and a challenge answerer. Proves domain control via DNS-01 (writing the `dns.txtrecord` seam duckdns publishes, verified against `8.8.8.8` — the NAT-friendly path) or HTTP-01 (in-memory token on port 80, only when spangap-web is in the build), selected by `s.acme.method`. Issued `tls_cert.pem`/`tls_key.pem` written as PEM files to the state store (not storage vars), picked up by spangap-net's `tls` server via `tlsReloadCert()`. The directory endpoint is hard-coded to Let's Encrypt production and not configurable; on a web-less build an explicit HTTP-01 fails loudly at renewal.

#### duckdns
A DuckDNS dynamic-DNS client keeping a stable public hostname pointed at the device's changing public IP (a single IPv4 A record for one subdomain) and publishing the DNS TXT record acme needs for DNS-01. On network-up it does one A-record update, then a cron entry refreshes every 15 minutes; if upnp has learned the gateway's external IP it is sent explicitly, otherwise DuckDNS infers it from the request source. Enablement is implicit — no enable switch; with `s.duckdns.domain` and `s.duckdns.token` (masked/write-only secret) both unset every entry point is a no-op. Advertises `dns.txtrecord.capable=1` at startup and subscribes to `dns.txtrecord`, publishing whatever acme writes as the subdomain's TXT record (empty value clears it; the TXT value is the ACME challenge string, not the token). Last-update IP and status are RAM-only, reset on reboot or network-down.

#### upnp
A UPnP IGD port-mapping client discovering the home router by SSDP and asking it (via SOAP) to map external ports to the device so its HTTPS listener and WebRTC data channel are reachable through NAT. Installs two mappings — HTTPS over TCP (`s.net.https_port`, default 443) and WebRTC over UDP (`s.net.webrtc_port`, default 4433) — with a 3600-second lease renewed every 15 minutes; the external TCP port is `s.upnp.ext_port` (0 reuses the HTTPS port). Starts automatically, re-installs on upstream-up/cron/`upnp update`, synchronously deletes all its mappings on upstream-down so it never leaves stale entries. Owns `s.upnp.enable` and `s.upnp.ext_port`; live mapping state RAM-only (via the `upnp` CLI). Deliberately does not configure IPv6, cannot defeat carrier-grade NAT (use a wg tunnel instead), has no socket/HTTP control surface. Paired with duckdns and acme it completes the remote-access stack, but the three are independent.

#### wg
Brings up a WireGuard tunnel from a spangap device to a single configured peer — client only: dials one remote endpoint, runs the Noise handshake and inline ChaCha20-Poly1305, and exposes a tunnel netif the device routes over. The remote-access alternative to upnp+duckdns+acme for devices behind CG-NAT or when the operator won't expose anything publicly; the tunnel core is a vendored/patched fork of `trombik/esp_wireguard` wrapped by `wg.cpp`. Not a task — `wgInit` registers net callbacks and starts the tunnel on `NET_EV_UPSTREAM_UP` (keying on upstream, not link, since it must reach the peer over the internet), tears down on upstream-down, restarts on config change, polls handshake state every 5s mirroring `wg.up`. Control via storage (`s.wg.*` settings, `secrets.wg.key` private key never sent to the browser, derived `s.wg.pubkey`) plus a `wg` CLI (status/up/down/keygen). Caveats: `s.wg.dns` is declared/shown but **not implemented** (no effect); `s.wg.address` is the local tunnel IP despite upstream's confusingly-named `allowed_ip` field; browser/TCP throughput is bounded (~1.3 Mbps at 100 ms RTT) while UDP passes through unthrottled and is the recommended high-bitrate path. Not a server, does not negotiate keys — matching keys are configured out of band.

#### sshd
Packages two independent SSH-2 functions in one straddle: a **server** (`sshd`) listening on TCP (default 22, up to two concurrent sessions) bridging each session's single channel into the device's `cli` or `log` services, and a **client** (`ssh`) dialing out to run a command or interactive shell. SSH-2 implemented directly against platform crypto (mbedTLS ChaCha20-Poly1305/X25519/SHA-256) with vendored Ed25519 (orlp) and ML-KEM-768 (mlkem-native); the server prefers post-quantum `mlkem768x25519-sha256` KEX with classical curve25519 fallback, the client offers classical curve25519 only. The server is a deliberately small surface — exactly one `session` channel supporting `shell`/`exec`/`subsystem log`, with forwarding, SFTP, X11, and agent forwarding all refused — authenticating via `ssh-ed25519` publickey against `s.sshd.authorized_keys` or password delegated to spangap-core's `auth` admin realm (username recorded but not enforced). Server storage under `s.sshd.*` and `secrets.sshd.host_seed`; client keeps `s.ssh.*`, `secrets.ssh.privkey`, and TOFU `s.ssh.known_hosts[]` (refusing changed host keys, naming the entry to clear). Starts automatically when built (`s.sshd.enabled` defaults on) but admits no one until a key is authorized or password set; `sshd-keygen`/`sshd-showkey` are top-level commands. Regenerating the host key invalidates all clients' known_hosts.

#### audio
A generic I2S audio engine plus an ITS packet server exposing one full-duplex audio device (single sample rate and block size) that streams the live mic to every connected client, mixes (sum + saturate) inbound audio from all clients into the speaker, and plays device-rate WAV files via `audioPlayWav(path)`. Reachable as browser DataChannel label `audio:1` (the task runs an ITS server on port 1) and uses a 12-byte `av_hdr_t`+payload wire format lifted verbatim from seccam so existing seccam decoders work unchanged; outbound headers carry real timestamps while inbound headers are optional (unparseable packets treated as raw device-rate PCM16, timestamps ignored). Clients pick an outbound codec — `pcm16`, `ulaw8k`, `ulaw16k`, `adpcm` — via the connect payload / DataChannel `protocol`, defaulting to `s.audio.codec` (`ulaw16k`). Owns the I2S read/write engine; most boards are pure Kconfig pin/mode data, with only control-bus input codecs (e.g. ES7210) needing an `audio_codec_ops_t` shim. Caveat: **no resampler** — WAV files and inbound audio must already be at the device rate. Storage vars under `s.audio.*` cover rate, block_ms, codec, gains, with ephemeral `audio.sample_rate`/`audio.clients` status keys.

### Board support

#### hw-tdeck
Board-support (HAL) straddle for the **LilyGo T-Deck Plus** (ESP32-S3FN16R8, 16 MB flash, 8 MB octal PSRAM, SX1262 LoRa, 320×240 ST7789V LCD, GT911 touch, trackball, ESP32-C3 QWERTY keyboard, GNSS, microSD). A **non-buildable** component that decides nothing about what the device does — it only makes the board usable, owning power/CS bring-up, the on-device input HAL, GNSS, optional RTC, battery monitor, and mic-codec shim, and publishing the board's pin map and tuning as Kconfig and storage keys. Contributes auto-dispatched hooks in two bands (`start:` before `spangapInit()` for bare-hardware prerequisites, `init:` after the platform is up); notably `tdeckStart` must precede `spangapInit()` because the SD mount needs the peripheral power rail up and shared-SPI CS lines parked HIGH. Pulls in spangap-lcd by default (drop with `--no-lcd`); documents a single shared FSPI bus and I2C0 bus with full pin maps; exposes `s.tdeck.*`/`s.gps.*` settings plus published battery/GPS telemetry. Caveats: flash size is intentionally not pinned (a 4 MB floor image self-grows to 16 MB); the original T-Deck is pin-compatible but the T-Deck **Pro** is not this straddle; hardware sub-revision quirks (GT911 address, ball orientation) in INTERNALS.md.

### Products & public profiles

#### reticulous (buildable)
A Reticulum / LXMF / Nomad Network hand-held device built on the spangap platform, porting the Reticulum stack and its messaging/text-web layers onto a single ESP32-S3 device with LoRa, WiFi, GPS, screen, and keyboard. This repo (`reticulous/reticulous`) is the **buildable**: it carries no firmware or board HAL, but ships the browser SPA, the LittleFS data image, and LCD launcher icons, and pulls in the whole reticulous straddle family plus the spangap platform. The org is decomposed into dual-side (firmware + browser) straddles — rns, the iface-* interfaces, lxmf, nomad, maps, and board HALs (hw-tdeck, hw-heltecv4) — and you build the buildable *with* a board (e.g. `--with spangap/hw-tdeck`), where every component auto-starts from build-generated init with no hand-written sequence. Primary target is the LilyGo T-Deck Plus (Heltec V4 headless secondary); PSRAM required. Notes the design has had security attention and is intended to be securable, but should not be relied on where a life depends on it being unhackable.

#### reticulous.github (org profile)
Public org-profile README for **Reticulous**, which implements the Reticulum mesh networking stack for ESP32 embedded boards with tools for LXMF messaging, Nomad micron browsing, and features not normally found on embedded Reticulum nodes (e.g. rnsh remote management). Built on and co-developed alongside the new **Spangap** Device Application Framework, serving both as a capable Reticulum node and a demonstration of Spangap's capabilities (multitasking, web UI, CLI, SSH, LCD apps). Ships ready-made hardware straddles for the Heltec-v4 LoRa board and the LilyGO T-Deck Plus; a generic build with no mesh radio still runs Reticulum over WiFi plus browser/CLI access and should work on any ESP32-S3 with PSRAM (ideally 4MB+ PSRAM, 8MB+ flash). Acknowledges documentation is still incomplete at this early stage; provides quick build/flash instructions requiring docker, python, git, curl.

#### spangap.github (org profile)
Profile README for **Spangap**, a Device Application Framework for ESP32 (currently ESP32-S3 with PSRAM) letting developers quickly build firmware with a reactive Vue/Quasar web interface and/or LVGL LCD touch UI, a unix-like CLI over serial/web/SSH, and a webserver with https/websockets/webrtc. Emphasises real RTOS multitasking without polling (via **ITS**, its Inter-Task Streaming library), a hierarchical storage object synchronizing between device (cJSON) and browser (Pinia), and a modular "straddle" architecture where each module bundles metadata, a browser NPM package, and an ESP-IDF component. Most of the README is a first-person origin narrative — how a network-camera project on the ESP32 grew into Spangap while solving ITS, the value store, and PSRAM/DRAM stack constraints — and cites Reticulous as its co-developed proof-of-concept. Apache 2.0; the author stresses this is early/alpha software (Status: June 2026), not done or stable, no community forum yet, contact rop@spangap.org. **Defect flagged:** the getting-started curl install command appears to contain a malformed URL (`.../spangap/spangap)`).

## Cross-cutting observations

These surfaced repeatedly across the summaries and are candidate reform items:

- **Consistent "does NOT own" sections** in the platform straddles (core, net, web, lcd, rnsh, upnp) are a strong existing convention worth standardising everywhere.
- **Two concrete defects to fix in-README:** the malformed curl URL in the spangap.github profile, and `s.wg.dns` documented-but-unimplemented in wg (plus `s.lxmf.auto_ticket` and `s.lcd.icon_res`, both documented-but-inert).
- **Stale-doc debt is already self-declared** in the build-system README (`s/` paths, aspirational layout boxes, obsolete `CLAUDE.md` files, hand-written `app_main()` sequences). This is the primary rewrite backlog.
- **INTERNALS.md / docs/ split** is the emerging pattern (operator guide in README, deep detail in INTERNALS.md or `docs/`). Not yet uniform across straddles.
- **Archived `README-old.old.md` copies** remain in four locations and should be retired once the reform lands.
